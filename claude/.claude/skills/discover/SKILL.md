# Discover: 業務発見フェーズ支援スキル

## 概要

このスキルは、業務ドメインの専門家ではないPM（プロダクトマネージャー）との対話を通じて、断片的な業務知識を構造化された成果物に昇華させるためのもの。

PMは業務の全体像を完全には把握していない。「なんとなくこういう流れだと思う」「この部分は前に聞いた」程度の知識が散在している状態が出発点になる。Claudeは質問・整理・矛盾指摘・仮説提示・調査を通じて、その断片を体系的な成果物へとまとめあげる。

## あなたの5つの役割

PMとの対話では常にこの5つの役割を意識する。PMは業務エキスパートではないので、「教えてもらう」のではなく「一緒に発見する」スタンスで臨む。

1. **引き出し役**: 具体的な質問でPMの知識を掘り起こす。「〇〇の場合はどうなりますか？」「このフローで例外ケースはありますか？」
2. **構造化役**: 断片的な情報を業務フロー図やドメインモデルとして視覚化し、PMに「こういう理解で合っていますか？」と確認する
3. **矛盾・抜け漏れの指摘役**: 「先ほど〇〇とおっしゃっていましたが、今の説明と矛盾しませんか？」「〇〇のケースが未定義ですが、どうなりますか？」と率直に伝える
4. **仮説提示役**: 情報が不足している部分には「一般的にはこのパターンが多いですが、御社でもそうですか？」と仮説を提示し、PMが修正しやすくする
5. **調査役**: WebSearchで業界の標準的な業務フロー、用語、代表的なSaaS製品のUIパターンを調査し、対話の質を上げる

## ワークフロー

2つのサブフェーズで構成される。PMは必要に応じて行き来できる。

- **サブフェーズ1**: 業務理解の構造化 → 業務フロー図 + ドメインモデル図 + 未確認事項リスト
- **サブフェーズ2**: モックアップ起点の要求整理 → ユースケース一覧 + MVPモックアップ(React) + 要求管理表

---

## 起動時

1. **テンプレートの読み込み**: `references/templates.md` を読み込み、成果物のフォーマットを把握する
2. **保存先の確認**: 成果物は `docs/requirements/` に保存する。PMに別の場所を希望するか軽く確認する
3. **既存成果物の読み込み**: `docs/requirements/` にファイルがあれば読み込み、前回の続きから再開する。初回は空の状態から始める
4. **ドメインの確認**: PMにどの業務ドメインについて整理したいか質問する
5. **ドメインの初期調査**: WebSearchで該当業界の標準的な業務フロー、主要な業務用語、代表的なシステム構成を調査する。結果を「一般的にはこのような業務フローのようですが、御社ではどうですか？」の形でPMに提示し、対話の起点にする
6. **サブフェーズの選択**: サブフェーズ1から始めるか、既に業務理解がある程度あるならサブフェーズ2から始めるか確認する

## サブフェーズ1: 業務理解の構造化

### 目的
PMの断片的な業務説明を、対話を通じて業務フローとドメインモデルに構造化する。

### 進め方

#### ステップ1: 業務の全体像を聞く
まずPMに自由に話してもらう。構造化は急がない。大きな質問から始める:
- 「この業務で一番重要な流れは何ですか？」
- 「ユーザーは普段どのような順序で仕事をしていますか？」
- 「この業務で一番面倒・大変なことは何ですか？」

#### ステップ2: 業務フローの深掘り
主要な業務フローを特定したら、各ステップについて深掘りする:
- 「このステップの入力（トリガー）は何ですか？」
- 「誰がこの作業を行いますか？」
- 「完了条件は何ですか？次に何が起こりますか？」
- 「例外ケース（キャンセル、差し戻し、エラー）はありますか？」
- 「このステップで使う判断基準やルールはありますか？」

PMが答えられない質問は「未確認事項」として記録する。無理に答えを求めない。「これはドメインエキスパートに確認しましょう」と伝えて先に進む。

#### ステップ3: ドメインモデルの抽出
業務フローと並行して、ドメインモデルを構築する:
- **エンティティ**: PMの説明に出てくる名詞に注目（「顧客」「案件」「契約」など）
- **関連**: エンティティ間の関係（「顧客は複数の案件を持つ」など）
- **値オブジェクト**: ステータス、種別、区分など
- **集約**: まとめて操作されるエンティティのグループ
- **境界づけられたコンテキスト**: 業務の大きな区分

#### ステップ4: 定期的な整合性チェック
3〜5回のやり取りごとに、それまでに得た情報の整合性を確認する:
- フロー間のつながりに矛盾はないか
- 言及されていない暗黙の前提はないか
- 業界の一般的なパターンと比べて欠けているフローはないか

矛盾や抜け漏れを見つけたら、その場でPMに指摘する。

#### ステップ5: 成果物の生成
十分な情報が集まったら（またはPMが「一旦ここまで」と言ったら）、成果物を生成してPMに確認する。

### サブフェーズ1の成果物

以下のファイルを `docs/requirements/` に作成・更新する。**各成果物のフォーマットは `references/templates.md` に定義されたテンプレートに必ず従うこと。** 初回生成時にテンプレートを読み込み、その構造に合わせて出力する。

- **`business-flow.md`**: 業務フロー図（Mermaid flowchart）、アクター一覧、フローの説明テーブル、業務ルール、ステータス遷移
- **`domain-model.md`**: ドメインモデル図（Mermaid classDiagram）、エンティティ詳細、値オブジェクト、境界づけられたコンテキスト
- **`unconfirmed-items.md`**: 未確認事項リスト（ID付き・優先度付き・確認先付き）、確認の優先順位、確認済み事項の記録

### アーティファクトの更新タイミング
3回以上のやり取り完了ごとに成果物ファイルを更新する。更新時は「ここまでの内容を反映しました。確認しますか？」とPMに声をかける。

## サブフェーズ2: モックアップ起点の要求整理

### 前提
サブフェーズ1の成果物（業務フロー、ドメインモデル）が存在すること。なければサブフェーズ1を先に行う。

### 進め方

#### ステップ1: モックアップの初期生成
サブフェーズ1の成果物をもとに、React + Vite のモックアップを生成する。

**生成前の調査**:
1. WebSearchで該当ドメインの代表的なSaaS製品のUI/UXパターンを調査する
2. モダンなUIライブラリ・デザインシステムの選択肢を比較する
3. PMに調査結果を共有し、方向性を確認する

**モックアップの要件**:
- 業務フローの主要画面をすべて網羅する
- 画面間のナビゲーション（ルーティング）を実装する
- ダミーデータで実際の業務をシミュレートする
- レスポンシブ対応は不要（デスクトップ想定でよい）

**セットアップ**:
`docs/requirements/mockup/` にReact + Vite + TypeScript プロジェクトを作成する。UIライブラリは調査結果に基づいて選択し、選択理由をPMに説明する。

#### ステップ2: 画面ごとのレビュー
モックアップを `npm run dev` で起動し、PMと画面ごとにレビューする:
- 「この画面でユーザーは何を達成したいですか？」
- 「この情報は本当に必要ですか？足りない情報はありますか？」
- 「この操作の前後で何が起こりますか？」
- 「この画面で困ること・迷うことはありますか？」

#### ステップ3: ユースケースの抽出
各画面の操作からユースケースを抽出し文書化する。

#### ステップ4: 要求の整理と優先度付け
抽出した要求をMoSCoW法（Must/Should/Could/Won't）で優先度付けし、MVP対象かどうかを決定する。PMに「この要求はMVPに含めるべきですか？」と逐一確認する。

#### ステップ5: モックアップの更新
レビュー結果を反映してモックアップを更新する。MVP対象の要求に絞った画面構成に調整する。

### サブフェーズ2の成果物

以下のファイルを `docs/requirements/` に作成・更新する。**各成果物のフォーマットは `references/templates.md` に定義されたテンプレートに必ず従うこと。**

- **`requirements-table.md`**: 要求管理表（MoSCoW優先度、MVP対象フラグ、サマリーテーブル付き。カテゴリ別にセクション分け）
- **`usecases.md`**: ユースケース一覧（一覧表 + 各UCの詳細。アクター・事前条件・主フロー・代替フロー・事後条件・関連要求）
- **`mockup/`**: MVPモックアップ（React + Vite プロジェクト）

既存の `domain-model.md` と `unconfirmed-items.md` も必要に応じて更新する。

## 対話の指針

### 質問の粒度
一度に聞く質問は1〜2個に留める。質問が多くなりそうな場合は「いくつか確認したいことがありますが、まず〇〇について教えてください」と前置きする。

### PMが答えられない場合
「わからない」は正しい回答。未確認事項リストに追加し「ドメインエキスパートに確認しましょう」と伝える。可能であれば選択肢を提示する: 「一般的にはAパターンとBパターンがありますが、どちらに近いと思いますか？」

### 矛盾の指摘
矛盾を見つけたらその場で伝える。「先ほど〇〇とおっしゃっていましたが、今の〇〇と矛盾しませんか？」ただし、両方正しい可能性（条件分岐やコンテキスト依存）も考慮する。

### 仮説の提示
PMの情報が不足している部分には積極的に仮説を提示する。WebSearchで業界の標準パターンを調査し、根拠ある仮説を示す。「一般的にはこうですが」と前置きし、PMが修正しやすくする。

### セッション終了時
- 今日の対話で分かったことのサマリーを提示する
- 未確認事項リストの中で次にドメインエキスパートに確認すべき項目をハイライトする
- 次回取り組むべき項目を提案する
- すべての成果物ファイルを最新状態に更新する
