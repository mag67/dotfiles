# AI駆動開発における機能要求の標準成果物を二層振る舞い仕様とする

## Context and Problem Statement

AI駆動開発において、設計・実装・検証の多くをAIに委ねる場合でも、人間は「何を実現し、どこに境界を引くか」を定める責務を持つ。
このとき、機能要求の成果物は、AIが参照する**主要な機能契約**であり、縦のガードレール（機能の検証体系）の立法に相当する。

一方で、機能要求の成果物だけで常に開発が完結するわけではない。実務上は、必要に応じて以下の補助成果物が併存しうる。

* API/Schema 契約（OpenAPI, GraphQL schema など）
* 非機能・構造制約（横のガードレール）
* 設計成果物（設計判断、データ設計、統合方針）
* 実行指示・運用指針（AGENTS.md, CI 設定, テスト戦略）

本決定が扱うのは、その中でも**縦のガードレールにおける機能要求の標準成果物**として、どの形式を採用するかである。

## Decision Drivers

* **AI実装可能性**: AIが意図を過不足なく解釈し、設計・実装に落とせること
* **AI検証可能性**: AIが成果物から受入観点・テスト観点を導出できること
* **人間監査性**: 非技術者を含むステークホルダーが読み、妥当性を判断できること
* **UI/UX変更耐性**: 画面や導線の変更で要求仕様がすぐに陳腐化しないこと
* **ビジネスルール分離性**: 決定論的ルールをシナリオ本文から切り離して管理できること
* **RED/GREEN運用適合性**: 自動検証・修正サイクルに接続しやすいこと
* **SSOT性**: 正本が明確で、派生ビューを生成できること
* **トークン効率**: AIのコンテキスト制約の中で扱いやすいこと
* **補助成果物との整合性**: 契約、設計、非機能制約と矛盾なく併用できること

## Considered Options

1. 二層振る舞い仕様（エッセンシャルシナリオ + ビジネスルールカタログ）
2. ユーザーストーリー + 受入基準
3. BDD / Gherkin（Given-When-Then）
4. ユースケース（Cockburn Fully Dressed）
5. 三層振る舞い仕様（上記1 + 常設の具体例マップ層）

## Decision Outcome

Chosen option: **二層振る舞い仕様**を、AI駆動開発における**機能要求の標準成果物**として採用する。

ただし、以下を明示する。

1. **本成果物は縦のガードレールにおける主要な機能契約である**
   ただし、API契約、設計成果物、横のガードレール、運用指示を排除しない。

2. **Layer 3（具体例マップ）は標準成果物には含めない**
   ただし、高リスク・高曖昧度領域では補助成果物として採用できる。

3. **AIの自律修復は前提ではなく目標特性である**
   本成果物は、AIが RED を検知し GREEN に到達しやすくするための構造を提供するが、自律修復の成功を常に保証するものではない。

## Decision Summary

### Layer 1: エッセンシャルシナリオ

UI要素や実装詳細を排除し、**ユーザーの意図**と**システムの責務**の対応で振る舞いを定義する。
事前条件・事後条件・拡張パスを持たせることで、AIが設計・E2E観点・異常系観点を導出しやすくする。

### Layer 2: ビジネスルールカタログ

決定論的なルールをシナリオ本文から分離し、**Wiegers 5類型**で整理する。
全ルールに **EARS 形式の要約**を持たせ、必要に応じて **Decision Table** または **数式 + Example Table** で詳細化する。

### Layer 3: 具体例マップ

**標準成果物からは外す。**
ただし、適格判定、課金、法令・契約解釈、境界条件の擦り合わせなど、曖昧性や監査要件が高い場合は、補助成果物として追加を許容する。

## Pros and Cons of the Options

### 1. 二層振る舞い仕様

Layer 1（エッセンシャルシナリオ）+ Layer 2（ビジネスルールカタログ）の二層構造。

* Good, because シナリオ（意味的な流れ）とビジネスルール（決定論的な判定）を分離できる
* Good, because Layer 2 の表・式から UT / property test / 受入観点を導出しやすい
* Good, because Layer 1 が UI 非依存であり、UI/UX リデザインに比較的強い
* Good, because 人間は EARS 要約から入り、必要時に表・式で詳細確認できる
* Good, because 成果物の種類を絞りつつ、AI と人間の双方に読みやすい
* Neutral, because UAT 導出精度は Layer 1 の事後条件の明確さに依存する
* Neutral, because 高曖昧度領域では補助的な具体例が別途必要になりうる
* Bad, because 既存の要求管理テンプレートとの整合にはカスタマイズが必要
* Bad, because 逆参照や関連図を手編集すると SSOT を損ないやすい

### 2. ユーザーストーリー + 受入基準

* Good, because 非技術者にも書きやすく、目的を共有しやすい
* Good, because Why を簡潔に表現できる
* Bad, because 単体では会話の出発点としての性質が強く、機能契約としては情報が不足しやすい
* Bad, because フローや分岐、シナリオ間依存を表現しにくい
* Bad, because ビジネスルールが受入基準に散在し、UT変換しにくい

### 3. BDD / Gherkin（Given-When-Then）

* Good, because 機械可読性と実行可能仕様への接続性が高い
* Good, because 監査性や外部説明責任が高い場面では有効
* Bad, because 本プロセスではステップ定義や保守コストが増分価値を上回りやすい
* Bad, because UI操作や具体手順に引き寄せられやすく、UI変更耐性が下がりやすい
* Bad, because ビジネスルールが多数のシナリオに重複しやすい

### 4. ユースケース（Cockburn Fully Dressed）

* Good, because 正常系・異常系・開始条件・終了条件を体系的に書ける
* Good, because テスト設計の素材として扱いやすい
* Bad, because 具体レベルで書くと UI や手順に依存しやすい
* Bad, because ビジネスルールが本文に埋め込まれやすい
* Bad, because 冗長になりやすく、トークン効率が下がる

### 5. 三層振る舞い仕様（二層 + 常設の具体例マップ）

* Good, because 具体例を固定資産として持てるため、合意形成や walkthrough に強い
* Good, because 受入例の説明責任が高い領域では有効
* Bad, because 常設前提にすると二重記述・三重記述を招きやすい
* Bad, because Layer 1 / Layer 2 の更新と同期し続ける運用コストが高い
* Bad, because すべての機能に常設する標準成果物としては過剰になりやすい

## 採用形式の詳細仕様

## Layer 1: エッセンシャルシナリオ

### 採用意図

Layer 1 は、UI・API・実装詳細ではなく、**利用者の意図とシステムの責務**を記述する。
これにより、画面変更や導線変更に対して比較的安定した機能要求表現を実現する。

### テンプレート

## SC-{ID}: {シナリオ名}

**ゴール**: {1文で記述}
**アクター**: {主アクター}
**事前条件**:

* {開始時に満たされるべき状態}

**事後条件**:

* {観測可能な状態変化 1}
* {観測可能な状態変化 2}

### 主フロー

| # | ユーザーの意図 | システムの責務         |
| - | ------- | --------------- |
| 1 | {意図}    | {責務}            |
| 2 | {意図}    | → [BR-xxx] {責務} |

### 拡張

* {n}a. {条件} → {振る舞い}

### 記述ルール

* **必須**: ゴール、アクター、事前条件、事後条件、主フロー
* **条件付き必須**: 拡張（例外・代替フローが存在する場合）
* **禁止**: ボタン名、画面部品名、URL、APIパス、DBテーブル名、クラス名などの実装詳細
* **推奨**: 事後条件は「観測可能な結果」に限定する
* **参照方法**: ビジネスルールは主フロー内で `BR-ID` をインライン参照する
* **正本**: Layer 1 から Layer 2 への参照を正本とし、逆参照一覧は生成物または補助情報として扱う

## Layer 2: ビジネスルールカタログ

### 採用意図

Layer 2 は、決定論的な判定・制約・計算を、シナリオ本文から切り離して管理する。
これにより、ルール単位の変更、テスト生成、レビュー、差分把握を容易にする。

### 分類体系

以下の **Wiegers 5類型**を採用する。

* 事実
* 制約
* 行為トリガー
* 推論
* 計算

### 自然言語記法

全ルールに **EARS 形式の要約**を付与する。
EARS はレビュー入口として使い、厳密性が必要な場合は表・式で詳細化する。

### 形式記法

* **制約 / 推論**: Decision Table
* **計算**: 数式 + Example Table
* **事実 / 行為トリガー**: EARS 要約を基本とし、必要時のみ補足記述

### テンプレート

## BR-{ID}: {ルール名}

**分類**: {事実 | 制約 | 行為トリガー | 推論 | 計算}

### ルール要約（EARS）

* **[{Ubiquitous | Event | State | Unwanted | Optional}]** {文}

### 詳細

* 制約 / 推論: Decision Table
* 計算: 数式 + Example Table
* 事実 / 行為トリガー: 必要に応じて補足

### 起源（任意）

* {法令 | 契約 | 社内規定 | ドメイン慣習 | 計算ロジック}

### 関連情報（任意）

* 関連ルール
* 補足注記
* 監査メモ

### 記述ルール

* **必須**: 分類、EARS 要約、必要な詳細表現
* **任意**: 起源、関連ルール、補足注記
* **非必須化**: 関連シナリオの逆参照一覧
  これは Layer 1 のインライン参照から生成可能なため、手編集の必須項目にしない
* **正本**: ルールの本文そのものを正本とし、一覧・依存図・逆参照は派生ビューとして扱う

## Layer 3: 具体例マップ

### 決定

Layer 3 は**標準成果物としては採用しない**。

### 理由

* Layer 2 に具体値付きの表がある場合、常設の例マップは重複しやすい
* 全機能で常設すると更新コストが増大する
* 具体例は AI がオンデマンド生成できる場合が多い

### 例外的に採用してよい条件

以下のいずれかに該当する場合、補助成果物として追加してよい。

* 法令、契約、課金、給与、会計など監査要求が強い
* 適格判定や境界条件の誤読リスクが高い
* ドメイン用語の解釈差が頻発する
* ステークホルダーとの walkthrough で具体例が継続的に必要
* AI 生成だけでは誤解が解消しない

## 外山氏ガードレールフレームワークとの対応

| ガードレール   | 本決定の対応                          | 検証方法            |
| -------- | ------------------------------- | --------------- |
| 縦のガードレール | Layer 1 + Layer 2 を主要な機能契約として採用 | テスト実行・受入観点検証    |
| 横のガードレール | 本決定のスコープ外                       | 静的解析・構造検査・非機能検証 |

補足として、縦のガードレールは本成果物だけで閉じるとは限らない。
契約テスト、API schema、property test、E2E シナリオなどの派生検証物は、本成果物から導出・連携されうる。

また、検証の配置は次の原則を採る。

* **pre-push**: 高速かつ決定的な検証（契約、ユニット、軽量なシナリオ検証）
* **CI**: より重い組合せ検証、E2E、広範な property test

AI は RED を検知した場合に修正提案または修復を試みるが、立法権と最終承認責任は人間が持つ。

## Consequences

### Positive Consequences

* 機能の流れと決定論的ルールを分離できる
* UI 変更に対して比較的安定した要求仕様になる
* AI がテスト観点や実装方針を導出しやすい
* 人間レビューでは EARS 要約を入口にしつつ、必要箇所だけ表・式を精査できる
* 正本と派生ビューを分けることで SSOT を維持しやすい

### Negative Consequences

* Jira や既存チケット運用にそのままは載せにくい
* Layer 1 の事後条件が弱いと AI の検証導出精度が落ちる
* 高曖昧度領域では、別紙の具体例や設計補助が必要になる
* 逆参照や依存関係を手運用すると保守負荷が上がる
* 導入初期は記法教育とテンプレート整備が必要

## Operational Rules

* **正本の優先順位**

  1. Layer 1 / Layer 2 本文
  2. 契約仕様・schema
  3. 派生テスト・逆参照一覧・依存図

* **変更原則**

  * 機能意図の変更は Layer 1 を起点に改定する
  * 決定論的判定の変更は Layer 2 を起点に改定する
  * 派生ビューは自動再生成を優先する

* **レビュー原則**

  * 人間は意図・境界・監査要件をレビューする
  * AI は実装差分、検証観点、表からのテスト化を支援する

## More Information

### 適用方針

この形式は、AI駆動開発における**標準形**として採用する。
ただし、以下は本決定の否定ではなく補助として併用可能である。

* OpenAPI / GraphQL schema などの契約
* 設計ADR、データモデル、統合仕様
* AGENTS.md や実装ガイド
* 補助的な具体例集

### 段階的成熟パス

| 段階  | ルール数の目安 | Layer 2 の記法                           | 構造化           |
| --- | ------: | ------------------------------------- | ------------- |
| 初期  |     〜30 | EARS中心                                | フラット管理        |
| 成長期 |  30〜100 | EARS + Decision Table / Example Table | 分類 + ディレクトリ分割 |
| 成熟期 |    100〜 | EARS + DMN準拠の表現を検討                    | 依存関係の可視化      |
